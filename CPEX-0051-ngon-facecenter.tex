\section{Abstract}
%% ═════════════════════════════════════════════════════════════════════════

SIDS Section~7.7 (\texttt{FlowSolution\_t}), Note~5, states that for
unstructured zones \texttt{GridLocation} is restricted to \texttt{Vertex}
or \texttt{CellCenter} unless a \texttt{PointList} or \texttt{PointRange}
is also present. The same restriction appears, near word-for-word, in
Section~12.4 (\texttt{DiscreteData\_t}). This forces every \texttt{FaceCenter}
solution on an unstructured zone to carry an explicit point set --- even
when the data spans literally every face in the zone, in which case the
point set is the identity permutation and conveys no information beyond
what is already implied by the zone's own element numbering.

For zones whose faces are described by \texttt{NGON\_n} elements, this
identity point set is redundant: SIDS Section~7.3 already guarantees that
\texttt{NGON\_n} elements ``specify all the faces in the grid,'' giving
faces the same kind of complete, contiguous, globally unique numbering
that cells already have --- which is precisely why \texttt{CellCenter}
never needed a \texttt{PointList}/\texttt{PointRange} in the first place.

This CPEX proposes a narrow, precisely scoped relaxation: when an
unstructured zone has \emph{complete NGON face coverage} (defined in
Section~\ref{sec:definition}), \texttt{GridLocation~=~FaceCenter} data may
be written and read \emph{without} a \texttt{PointList}/\texttt{PointRange},
indexed directly against the zone's \texttt{NGON\_n} element numbering,
exactly as \texttt{CellCenter} is indexed against the zone's
\texttt{CellDimension}-order elements. The relaxation is strictly additive:
files using the existing identity-point-set workaround remain fully
conformant, and the change is scoped to \texttt{NGON\_n} coverage only ---
it explicitly does \emph{not} extend to classical 2-D element sections
(\texttt{TRI\_3}, \texttt{QUAD\_4}, \ldots), which are conventionally used
for boundary patches only and do not carry SIDS's ``all faces'' guarantee.

While researching the current behavior, two independent implementation
gaps were found that this CPEX's reference implementation also addresses:
\texttt{cg\_sol\_write} cannot create a \texttt{FaceCenter}-located
solution directly (Section~\ref{sec:cg_sol_write_gap}), and
\texttt{cgnscheck} never reads a \texttt{FlowSolution\_t}'s
\texttt{PointList}/\texttt{PointRange} when validating data-array sizes
(Section~\ref{sec:cgnscheck_gap}), so today's identity-point-set files are
not actually verifiable by the reference conformance tool.

\section{Motivation and Rationale}
%% ═════════════════════════════════════════════════════════════════════════

\subsection{Problem Statement}
\label{sec:problem-statement}

A common CFD data-storage need is a quantity defined on \emph{every} face
of an unstructured mesh (e.g., face-normal velocities, surface fluxes on
all element boundaries). For a zone built from \texttt{NFACE\_n}/\texttt{NGON\_n}
polyhedral connectivity, the following tree is representative:

\begin{lstlisting}[language=C]
fluid_zone Zone_t [[27  8  0]]
+---ZoneType ZoneType_t "Unstructured"
+---NGonElements Elements_t [22  0]
|   +---ElementRange IndexRange_t [ 1 36]
|   :
+---NFaceElements Elements_t [23  0]
|   +---ElementRange IndexRange_t [37 44]
|   :
+---CellData FlowSolution_t
|   +---GridLocation GridLocation_t "CellCenter"
|   +---Pressure DataArray_t   /* 8 values -- no PointList/PointRange */
+---FaceData FlowSolution_t
    +---GridLocation GridLocation_t "FaceCenter"
    +---NormalX DataArray_t     /* 36 values */
    :
    +---PointRange IndexRange_t [[1, 36]]
        ^^^^ identity range -- required by Note 5, conveys no new information
\end{lstlisting}

\texttt{CellData} needs no point set: \texttt{GridLocation~=~CellCenter}
alone is sufficient, because the zone's cells are already uniquely and
completely numbered by its volume \texttt{Elements\_t} sections.
\texttt{FaceData}, however, is required by SIDS Note~5 to carry a
\texttt{PointRange}, even though \texttt{[1,36]} is exactly the
\texttt{NGonElements} section's own \texttt{ElementRange} --- information
the reader already has. Two consequences follow:

\begin{enumerate}[leftmargin=2em]
\item \textbf{Confusion.} Users and tool authors reasonably ask why an
  identity point set is mandatory for one \texttt{GridLocation} and
  forbidden (well, unnecessary) for another, when both are governed by the
  same element-numbering machinery.
\item \textbf{Detection cost.} Any code path that wants to special-case
  ``this solution spans every face'' --- to avoid gather/scatter
  overhead in a parallel read, for instance --- must first read the point
  set and compare it against the face count, on every zone, for every
  solution. This is a small but repeated and entirely avoidable
  per-solution check.
\end{enumerate}

\subsection{Origin of the Restriction}

Note~5 was introduced by CPEX~0031 (``revamped usage of \texttt{PointList}
and \texttt{PointRange}''), incorporated into SIDS~v3.1. The SIDS gives
its own rationale immediately following the notes list in
Section~7.7:

\begin{quote}\small\itshape
``For unstructured grids, the value of \texttt{GridLocation} alone
specifies location and indexing of flow solution data only for vertex
and cell-centered data. The reason for this is that element-based grid
connectivity provided in the \texttt{Elements\_t} data structures
explicitly indexes only vertices and cells. For data stored at alternate
grid locations (e.g., edges), additional connectivity information is
needed. This is provided by the optional fields \texttt{PointRange} and
\texttt{PointList}\ldots''
\end{quote}

This is a correct and still-necessary rule for the \emph{general} case:
an unstructured zone is only ever \emph{required} to number its vertices
and its \texttt{CellDimension}-order elements. A zone built purely from
\texttt{TETRA\_4}/\texttt{PYRA\_5}/\texttt{PENTA\_6}/\texttt{HEXA\_8}/
\texttt{MIXED} sections, with no face-type \texttt{Elements\_t} section
at all, has no face numbering whatsoever --- \texttt{FaceCenter} is
genuinely meaningless there without an explicit point set.

Notably, SIDS~v3.1 is also the version in which \texttt{NFACE\_n} was
formalized and the \texttt{NGON\_n} description was substantially revised
(per the same changelog entry). The two changes were made in the same
release but were not cross-referenced: Note~5 was written for, and still
correctly describes, the general classical-element case, but it was never
revisited to carve out the \texttt{NGON\_n} case where a complete face
numbering \emph{is} guaranteed to exist.

\subsection{Why \texttt{NGON\_n} Changes the Picture}
\label{sec:why-ngon}

SIDS Section~7.3 (\texttt{Elements\_t}) states two facts that together
give faces the same numbering guarantee that cells already enjoy:

\begin{quote}\small\itshape
``The elements are indexed with a global numbering system, starting at 1,
for all element sections under a given \texttt{Zone\_t} data structure.
The global numbering insures that each element, whether it's a cell, a
face, or an edge, is uniquely identified by its number. They are also
listed as a continuous list of element numbers within any single element
section.''
\end{quote}

\begin{quote}\small\itshape
``Arbitrary polyhedral elements may be defined using the \texttt{NGON\_n}
and \texttt{NFACE\_n} element types. \textbf{The \texttt{NGON\_n} element
type is used to specify all the faces in the grid}, and the
\texttt{NFACE\_n} element type is then used to define the polyhedral
elements as a collection of these faces.''
\end{quote}

Combined, these two statements mean that whenever a zone's volume cells
are described with \texttt{NFACE\_n}/\texttt{NGON\_n} connectivity, the
union of that zone's \texttt{NGON\_n} \texttt{ElementRange}(s) is --- by
the SIDS's own existing guarantees, with no new machinery required --- a
complete, contiguous, globally unique numbering of every face in the
zone. This is structurally identical to how a zone's volume
\texttt{Elements\_t} sections already give \texttt{CellCenter} an
implicit, unambiguous numbering.

\subsection{Precedent: \texttt{CellCenter} Already Works This Way}

SIDS Section~7.7 defines the \texttt{DataSize[]} structure function,
which is the authoritative statement of how \texttt{FlowSolution\_t}
array sizes are derived. Quoted verbatim from the current SIDS source
(non-\texttt{Rind} branches; the \texttt{Rind}-present branch follows the
same pattern):

\begin{lstlisting}[language=C]
if (PointRange/PointList is present) then
  {
  DataSize[] = ListLength[] ;
  }
else if (Rind is absent) then
  {
  if (GridLocation = Vertex) or (GridLocation is absent)
    {
    DataSize[] = VertexSize[] ;
    }
  else if (GridLocation = CellCenter) then
    {
    DataSize[] = CellSize[] ;
    }
  }
\end{lstlisting}

The \texttt{CellCenter} branch already relies \emph{entirely} on implicit
element-based sizing: no \texttt{PointList}/\texttt{PointRange} is
required there, because \texttt{CellSize[]} is already well-defined from
the zone's \texttt{CellDimension}-order \texttt{Elements\_t} sections. This CPEX asks for nothing more than
extending that same, already-accepted pattern to \texttt{FaceCenter} in
the one case where an equally strong SIDS guarantee already exists ---
\texttt{NGON\_n} coverage. See Section~\ref{sec:datasize} for the precise
proposed addition to \texttt{DataSize[]}.

\subsection{Existing Implementation Gaps}
\label{sec:implementation-gaps}

Two related gaps in the reference implementation were found while
preparing this proposal. Neither is created by this CPEX, but both must
be fixed as part of its reference implementation, since they sit
directly in the code paths this CPEX touches.

\subsubsection{\texttt{cg\_sol\_write} rejects plain \texttt{FaceCenter}}
\label{sec:cg_sol_write_gap}

\texttt{cg\_sol\_write} (\texttt{cgnslib.c}) validates its
\texttt{location} argument against an explicit allow-list:

\begin{lstlisting}[language=C]
if (location != CGNS_ENUMV(Vertex) &&
    location != CGNS_ENUMV(CellCenter) &&
    location != CGNS_ENUMV(IFaceCenter) &&
    location != CGNS_ENUMV(JFaceCenter) &&
    location != CGNS_ENUMV(KFaceCenter)) {
    cgi_error("Given grid location not supported for FlowSolution_t");
    return CG_ERROR;
}
\end{lstlisting}

Plain \texttt{FaceCenter} --- the only location an unstructured zone can
use for face data --- is not in this list. This does \emph{not} block the
\emph{explicit}-point-set form in Example~\ref{sec:example1}:
\texttt{cg\_sol\_ptset\_write()} already creates a \texttt{FaceCenter}
solution with an accompanying \texttt{PointList}/\texttt{PointRange} in a
single call today, because it validates via \texttt{cgi\_check\_location}
directly and, internally, works around \texttt{cg\_sol\_write}'s allow-list
by calling it with \texttt{location = Vertex} and overwriting
\texttt{sol->location} afterward --- itself telling evidence that the
allow-list is redundant with \texttt{cgi\_check\_location} rather than an
independent rule (see Section~\ref{sec:mll_sol_write_fix}). The gap that
\emph{does} block today is the \emph{point-set-free} form this CPEX
introduces: \texttt{cg\_sol\_ptset\_write()} requires \texttt{npnts~>~0},
so it cannot be used to create a bare \texttt{FaceCenter} solution with no
point set at all, and \texttt{cg\_sol\_write} is the only entry point for
that shape --- which is exactly where the allow-list rejects it. Today,
the only way to create a point-set-free \texttt{FaceCenter} solution is
the two-step sequence \texttt{cg\_sol\_write(..., CellCenter, ...)}
followed by \texttt{cg\_gridlocation\_write(..., FaceCenter)}, which
happens to work only because \texttt{cg\_gridlocation\_write} performs its
own, separate validation (\texttt{cgi\_check\_location}) that \emph{does}
permit \texttt{FaceCenter} for \texttt{CellDimension~$\geq$~3}. This is a
latent usability bug independent of Note~5, and it directly affects every
user who wants to write a point-set-free \texttt{FaceCenter} solution
today (e.g.\ the identity-\texttt{PointRange} workaround plus later
\texttt{cg\_gridlocation\_write} override, or any future use of this
CPEX's implicit form), whether or not this CPEX is adopted. Tracked as
\href{https://github.com/CGNS/CGNS/issues/964}{CGNS/CGNS \#964}.

\subsubsection{\texttt{cgnscheck} never reads \texttt{FlowSolution\_t}'s point set}
\label{sec:cgnscheck_gap}

\texttt{check\_solution()} (\texttt{cgnscheck.c}) computes the expected
data-array size via \texttt{get\_data\_size()}, which contains:

\begin{lstlisting}[language=C]
if (z->type == CGNS_ENUMV(Unstructured)) {
    error ("grid location %s not valid for unstructured zone",
        cg_GridLocationName (location));
    return 0;
}
\end{lstlisting}

This branch is checked \emph{before} \texttt{get\_data\_size()} ever looks
at \texttt{GridLocation~=~FaceCenter}, so it fires for \emph{any}
\texttt{GridLocation} other than \texttt{Vertex}/\texttt{CellCenter} on an
unstructured zone --- including \texttt{FaceCenter} with a fully
populated, conformant \texttt{PointRange}. \texttt{check\_solution()}
never calls \texttt{cg\_ptset\_info}/\texttt{cg\_ptset\_read} at all, so
today's required identity-point-set workaround
(Section~\ref{sec:problem-statement}) is not merely \emph{unvalidated} by
the reference conformance checker --- it is \emph{actively misreported}.
\texttt{error()} is non-fatal but not silent: it prints a counted
\texttt{ERROR:} line and increments \texttt{cgnscheck}'s error tally for
every \texttt{FaceCenter} solution on every unstructured zone in the file,
today, for files that are fully conformant under the \emph{current} SIDS.
This is a stronger motivation than mere non-validation: it is a
false-positive in the reference tool that this CPEX's fix also happens to
resolve. This CPEX's reference implementation fixes
\texttt{get\_data\_size()} to honor \texttt{ListLength[]} from an explicit
point set (already required by the current SIDS) in addition to the new
implicit \texttt{NGON\_n} sizing this CPEX adds. Tracked as
\href{https://github.com/CGNS/CGNS/issues/965}{CGNS/CGNS \#965}.

\section{Detailed Description}
%% ═════════════════════════════════════════════════════════════════════════

\subsection{Design Overview}

The proposal has four parts:

\begin{enumerate}[leftmargin=2em]
\item A formal SIDS definition of \emph{complete NGON face coverage}
  (Section~\ref{sec:definition}), built entirely from existing SIDS
  guarantees --- no new node types or structural elements are
  introduced.
\item A redline amendment to SIDS Note~5 in \texttt{FlowSolution\_t}
  (\S7.7) and the matching clause in \texttt{DiscreteData\_t}
  (\S12.4), and a corresponding addition to the \texttt{DataSize[]}
  structure function (Section~\ref{sec:datasize}).
\item MLL changes (Section~\ref{sec:mll}) to compute and validate the
  implicit face count, expose it to readers via a new public accessor
  (Section~\ref{sec:mll_helper}), and fix the two unrelated
  implementation gaps from Section~\ref{sec:implementation-gaps} (as
  standalone, independently backportable patches --- see
  Section~\ref{sec:api-summary}).
\item A test plan (Section~\ref{sec:test-plan}) covering the coverage
  predicate, the new sizing rule, and the write-ordering constraint this
  CPEX introduces (Section~\ref{sec:backward-compat}).
\end{enumerate}

\subsection{Definition: Complete NGON Face Coverage}
\label{sec:definition}

An unstructured zone $Z$ has \textbf{complete NGON face coverage} if and
only if:

\begin{enumerate}[leftmargin=2em]
\item $Z$ contains one or more \texttt{Elements\_t} sections of type
  \texttt{NGON\_n}; and
\item the union of the \texttt{ElementRange}s of those sections forms a
  single contiguous block $[E_{\mathrm{start}}, E_{\mathrm{end}}]$ of
  the zone's global element numbering (trivially true for one section;
  for multiple \texttt{NGON\_n} sections, their ranges must partition
  $[E_{\mathrm{start}}, E_{\mathrm{end}}]$ with no gaps, and no element
  index inside that block may belong to a non-\texttt{NGON\_n}
  section).
\end{enumerate}

When these conditions hold, the face count is
$N_{\mathrm{face}} = E_{\mathrm{end}} - E_{\mathrm{start}} + 1$, and the
\textbf{implicit \texttt{FaceCenter} indexing rule} is: the $j$-th entry
(1-based) of the \texttt{DataArray\_t} maps to global element index

\[ E_{\mathrm{start}} + j - 1, \qquad 1 \le j \le N_{\mathrm{face}}. \]

This rule refers \emph{only} to the $[E_{\mathrm{start}},
E_{\mathrm{end}}]$ block; it says nothing about, and is unaffected by,
any other \texttt{Elements\_t} sections elsewhere in $Z$'s global
numbering. In particular, a zone may freely contain separate
boundary-patch sections (\texttt{TRI\_3}, \texttt{QUAD\_4}, \ldots) used
for \texttt{BC\_t} definitions \emph{outside} the \texttt{NGON\_n} block
--- a common pattern in files where the volume mesh is polyhedral but
boundary patches are still exposed via classical elements for tool
compatibility. Their presence does not disqualify $Z$ from complete NGON
face coverage and does not change what the implicit indexing rule means;
those sections simply are not candidates for it. This is a deliberate
relaxation from an earlier draft of this CPEX, which excluded any
zone containing such sections; that exclusion turned out to be
unnecessary, since \texttt{TRI\_3}/\texttt{QUAD\_4} sections were never
valid candidates for ``all faces'' in the first place (see below) ---
the only thing that matters is that $[E_{\mathrm{start}},
E_{\mathrm{end}}]$ itself is unambiguous, which condition~2 already
guarantees regardless of what else exists in the zone.

This CPEX intentionally does \textbf{not} extend the implicit-sizing idea
to classical 2-D sections (\texttt{TRI\_3}, \texttt{QUAD\_4}, \ldots)
themselves, i.e., there is no equivalent rule that lets \emph{their}
combined \texttt{ElementSize} stand in for ``the face count of the
zone.'' Unlike \texttt{NGON\_n}, the SIDS gives no guarantee that such
sections enumerate \emph{all} faces of a zone; in the overwhelming
majority of existing CGNS files they enumerate boundary faces only.
Treating their combined \texttt{ElementSize} as ``the face count of the
zone'' would silently misinterpret a large fraction of legacy files.
This CPEX similarly does not extend to \texttt{EdgeCenter} on 2-D zones:
SIDS has no analogous ``this element type enumerates all edges''
primitive for edges, so no equivalent guarantee is available there.

\subsection{Revised SIDS Text: \texttt{FlowSolution\_t} (\S7.7, Note~5)}
\label{sec:flowsolution-redline}

\textbf{Current text:}
\begin{sidsold}
For unstructured zones GridLocation options are limited to Vertex or
CellCenter, unless one of PointList or PointRange is present.
\end{sidsold}

\textbf{Proposed text:}
\begin{sidsnew}
For unstructured zones GridLocation options are limited to Vertex or
CellCenter, unless one of PointList or PointRange is present,
{\color{cgnsgold}or unless GridLocation is FaceCenter and the zone has
complete NGON face coverage (\S7.3), in which case indexing follows the
zone's NGON\_n element numbering directly.}
\end{sidsnew}

\subsection{Revised SIDS Text: \texttt{DiscreteData\_t} (\S12.4)}
\label{sec:discretedata-redline}

The matching clause in \texttt{DiscreteData\_t} is amended the same way,
for consistency --- both notes were introduced together by CPEX~0031 and
have always read the same way, modulo word order (``PointRange or
PointList'' here vs.\ ``PointList or PointRange'' in \S7.7):

\textbf{Current text:}
\begin{sidsold}
For unstructured zones GridLocation options are limited to Vertex or
CellCenter, unless one of PointRange or PointList is present.
\end{sidsold}

\textbf{Proposed text:}
\begin{sidsnew}
For unstructured zones GridLocation options are limited to Vertex or
CellCenter, unless one of PointRange or PointList is present,
{\color{cgnsgold}or unless GridLocation is FaceCenter and the zone has
complete NGON face coverage (\S7.3), in which case indexing follows the
zone's NGON\_n element numbering directly.}
\end{sidsnew}

\texttt{ZoneSubRegion\_t} and \texttt{BC\_t}, the other two structures
touched by CPEX~0031, are deliberately \emph{not} amended here.
\texttt{ZoneSubRegion\_t} always requires an explicit region descriptor
(point set, \texttt{BCRegionName}, or \texttt{GridConnectivityRegionName})
regardless of \texttt{GridLocation}, so Note~5 there is not a burden in
the same way. \texttt{BC\_t} does not carry this clause at all (a BC
patch is definitionally a \emph{subset}, so an explicit point set is
appropriate there even for a complete-coverage zone). See
Section~\ref{sec:zonesubregion-scope} for discussion of whether a future
CPEX should revisit \texttt{ZoneSubRegion\_t} as well.

\subsection{Revised \texttt{DataSize[]} Structure Function}
\label{sec:datasize}

Section~7.7's \texttt{DataSize[]} definition gains one new
\texttt{FaceCenter} case inside \emph{each} of the two existing
\texttt{Rind} branches, in direct parallel with the existing
\texttt{Vertex} and \texttt{CellCenter} cases. The explicit
\texttt{PointRange}/\texttt{PointList} branch remains first and still
takes precedence, so existing identity-point-set files are unaffected;
and because the new cases sit \emph{inside} the \texttt{Rind} branches
(rather than ahead of them), a \texttt{FaceCenter} solution carrying rind
elements is still sized with its rind contribution, exactly as
\texttt{CellCenter} is. The redline below follows the existing SIDS
pseudocode style (\texttt{then} keywords, braces on their own lines)
exactly as it appears in the current SIDS source, so the amendment can be
applied to the standard's text verbatim; the two new cases are marked:

\begin{lstlisting}[language=C]
if (PointRange/PointList is present) then
  {
  DataSize[] = ListLength[] ;
  }
else if (Rind is absent) then
  {
  if (GridLocation = Vertex) or (GridLocation is absent)
    {
    DataSize[] = VertexSize[] ;
    }
  else if (GridLocation = CellCenter) then
    {
    DataSize[] = CellSize[] ;
    }
  else if (GridLocation = FaceCenter) and              /* new */
          (zone has complete NGON face coverage) then
    {
    DataSize[] = FaceSize[] ;
    }
  }
else if (Rind is present) then
  {
  if (GridLocation = Vertex) or (GridLocation is absent) then
    {
    DataSize[] = VertexSize[] + [a + b,...] ;
    }
  else if (GridLocation = CellCenter)
    {
    DataSize[] = CellSize[] + [a + b,...] ;
    }
  else if (GridLocation = FaceCenter) and              /* new */
          (zone has complete NGON face coverage) then
    {
    DataSize[] = FaceSize[] + [a + b,...] ;
    }
  }
\end{lstlisting}

Here \texttt{FaceSize[]} is defined as
$E_{\mathrm{end}} - E_{\mathrm{start}} + 1$ for the zone's contiguous
\texttt{NGON\_n} block (Section~\ref{sec:definition}), and
\texttt{RindPlanes = [a,b,...]} as in the existing definition. The name
\texttt{FaceSize[]}, rather than \texttt{NFace[]}, is chosen to avoid
confusion with \texttt{NFACE\_n} cell counts.

Note that \S12.4 requires no matching \texttt{DataSize[]} amendment: the
SIDS defines \texttt{DiscreteData\_t}'s \texttt{DataSize[]} by reference
--- ``It is identical to the function \texttt{DataSize[]} defined for
\texttt{FlowSolution\_t}'' --- so the change above propagates to
\texttt{DiscreteData\_t} automatically. Only the \S12.4 \emph{Note}
(Section~\ref{sec:discretedata-redline}) needs its own redline.

\subsection{MLL Changes}
\label{sec:mll}

\subsubsection{New internal helper}
\label{sec:mll_helper}

\begin{lstlisting}[language=C]
/* cgns_internals.c */
int cgi_zone_ngon_face_range(cgns_zone *zone,
                              cgsize_t *start, cgsize_t *end);
    /* Returns CG_OK and the contiguous [start,end] face range if the
     * zone has complete NGON face coverage per the definition in
     * this document's Detailed Description section; returns
     * CG_NODE_NOT_FOUND if the zone has no NGON_n sections at all;
     * returns CG_ERROR (with cgi_error() detail) if NGON_n sections
     * exist but coverage is not complete (gaps, or mixed with
     * classical 2-D face-bearing sections). The cgi_error() message
     * names the zone and, for the gap case, the interrupting section
     * (e.g. "zone 'fluid_zone': NGON_n sections [1,20] and [31,50]
     * are not contiguous (elements 21-30 belong to section 'Mixed'"),
     * so the failure is diagnosable without re-deriving element
     * ranges by hand. Rejects (CG_ERROR) any ElementRange with
     * start < 1 or end < start, and any face count that would
     * overflow cgsize_t arithmetic in the caller's size computation,
     * before returning -- both ranges are read from the file and
     * must not be trusted blindly. */
\end{lstlisting}

This helper is used by \texttt{cg\_sol\_write}/\texttt{cg\_field\_write}
(and their \texttt{cg\_discrete\_*} counterparts) whenever
\texttt{GridLocation~=~FaceCenter} is requested on an unstructured zone
with no accompanying point set, to (a)~confirm the relaxation applies and
(b)~compute the expected \texttt{DataArray\_t} size for validation on
write, exactly as \texttt{CellSize[]} is already used for
\texttt{CellCenter}. Coverage is a property of the zone's
\texttt{Elements\_t} section layout, already resident in memory as
\texttt{zone->section[]}; the reference implementation computes it once
per zone (caching the result on \texttt{cgns\_zone}) rather than
re-scanning the section list on every \texttt{cg\_field\_write} call to a
multi-field \texttt{FaceCenter} solution, and invalidates the cached
result on \texttt{cg\_section\_write}/\texttt{cg\_section\_partial\_write}.

A thin public wrapper is also added, so that reader-side code (as in
Example~\ref{sec:example3}) does not have to reimplement the coverage
predicate independently for every downstream tool:

\begin{lstlisting}[language=C]
/* cgnslib.c; public API */
int cg_zone_ngon_face_range(int fn, int B, int Z,
                             cgsize_t *start, cgsize_t *end);
    /* Thin wrapper over cgi_zone_ngon_face_range() for read-side
     * consumers. Same return-value contract. */
\end{lstlisting}

\subsubsection{Fix: \texttt{cg\_sol\_write} allow-list}
\label{sec:mll_sol_write_fix}

Rather than add \texttt{FaceCenter} as one more literal in
\texttt{cg\_sol\_write}'s local allow-list, the reference implementation
\emph{removes} that allow-list and delegates to
\texttt{cgi\_check\_location()} --- the same dimension- and zone-type-aware
check that \texttt{cg\_gridlocation\_write} and \texttt{cg\_sol\_ptset\_write}
already trust, and the reason those two entry points already accept
\texttt{FaceCenter} correctly today while \texttt{cg\_sol\_write} does not.
\texttt{cg\_sol\_write}'s own literal allow-list and its separate,
hand-written \texttt{[IJK]FaceCenter}/\texttt{Structured}-only check
are two independent copies of a rule \texttt{cgi\_check\_location} already
enforces; keeping both is exactly the kind of duplication that let them
drift apart and produced this bug in the first place. Because
\texttt{cgi\_check\_location} needs the zone's \texttt{CellDimension} and
\texttt{ZoneType\_t}, the check must move to \emph{after} the zone lookup
(\texttt{cg\_sol\_write} currently validates \texttt{location} before
\texttt{cgi\_get\_file}/\texttt{cgi\_get\_zone} are called):

\begin{lstlisting}[language=C]
/* was: a literal allow-list checked before the zone lookup, plus a
 * separate structured-only [IJK]FaceCenter check further down. Both
 * are replaced by one delegated call, placed after the zone lookup: */
zone = cgi_get_zone(cg, B, Z);
if (zone == 0) return CG_ERROR;
if (cgi_check_location(cg->base[B-1].cell_dim, zone->type, location))
    return CG_ERROR;   /* cgi_check_location has already set cgi_error() */
\end{lstlisting}

This is strictly additive in effect: every \texttt{location} value
previously accepted by the old allow-list is still accepted (each is one
of \texttt{cgi\_check\_location}'s \texttt{CG\_OK} cases), plain
\texttt{FaceCenter} is now accepted for \texttt{CellDimension~$\geq$~3}
regardless of zone type (matching what \texttt{cg\_gridlocation\_write}
already permits today), and \texttt{[IJK]FaceCenter} on a non-\texttt{Structured}
zone is still rejected, with the same error message, by
\texttt{cgi\_check\_location} itself. This fix is independent of the rest
of this CPEX (it does not by itself relax Note~5) but is required in order
for Example~\ref{sec:example2}'s single-call write to work; without it,
applications must still fall back to the two-step
\texttt{cg\_sol\_write}~+~\texttt{cg\_gridlocation\_write} sequence for the
point-set-free form. Being a narrowly-scoped, independently testable bug
fix with no dependency on the NGON-coverage logic, it can land as its own
patch against the current release line without waiting on the rest of
this CPEX; see the note in Section~\ref{sec:api-summary}.

\subsubsection{Fix: \texttt{cgnscheck} point-set-aware sizing}

\texttt{get\_data\_size()} (Section~\ref{sec:cgnscheck_gap}) is updated
to: (1)~read a \texttt{PointList}/\texttt{PointRange} under
\texttt{FlowSolution\_t}/\texttt{DiscreteData\_t}, if present, and use its
\texttt{ListLength} as the expected size (this branch is required by the
\emph{current} SIDS and was simply never implemented); and (2)~when no
point set is present and \texttt{GridLocation~=~FaceCenter} on an
unstructured zone, call \texttt{cgi\_zone\_ngon\_face\_range()} and use
the returned count. Only when neither applies does
\texttt{get\_data\_size()} retain its current ``grid location not valid
for unstructured zone'' error.

\subsubsection{No change to on-disk representation}

No new node type, attribute, or data layout is introduced. A conformant
file under this CPEX is a \texttt{FlowSolution\_t} (or
\texttt{DiscreteData\_t}) node with \texttt{GridLocation~=~FaceCenter}
and \emph{no} \texttt{PointList}/\texttt{PointRange} child, on a zone with
complete NGON face coverage. Because this is a strict superset of what
Note~5 currently forbids, this CPEX changes only the \emph{normative
interpretation} of an existing, legal-today node shape --- it does not
add anything a pre-CPEX HDF5/ADF reader would fail to open at the
byte level.

\subsection{Scope / Non-Goals}

\begin{itemize}[leftmargin=2em]
\item \textbf{Not addressed:} \texttt{EdgeCenter} on 2-D unstructured
  zones. No SIDS element type gives edges a completeness guarantee
  analogous to \texttt{NGON\_n}'s for faces.
\item \textbf{Not addressed:} implicit sizing for classical 2-D sections
  (\texttt{TRI\_3}/\texttt{QUAD\_4}/\ldots). See
  Section~\ref{sec:definition}.
\item \textbf{Not addressed:} \texttt{ZoneSubRegion\_t}, \texttt{BC\_t}.
  See Section~\ref{sec:zonesubregion-scope}.
\item \textbf{Not required:} no change to \texttt{NFACE\_n}'s own
  definition, or to how \texttt{CellCenter} is already handled; both are
  unaffected and serve as the existing precedent this CPEX generalizes.
\end{itemize}

%% ═════════════════════════════════════════════════════════════════════════
\section{Backward Compatibility}
\label{sec:backward-compat}
%% ═════════════════════════════════════════════════════════════════════════

\begin{itemize}[leftmargin=2em]
\item \textbf{Existing files:} Fully compatible. Files using the current
  identity-point-set workaround remain valid; the explicit
  \texttt{PointRange}/\texttt{PointList} branch of the revised
  \texttt{DataSize[]} function still takes precedence over the new
  implicit branch, so nothing about their interpretation changes.

\item \textbf{Existing readers (pre-CPEX-0051).} A pre-CPEX reader that
  encounters a \texttt{FaceCenter} \texttt{FlowSolution\_t} with no point
  set on a complete-NGON zone will, under the \emph{old} SIDS text,
  consider the file non-conformant (Note~5 as currently written requires
  the point set), and a naive reader may simply misinterpret the array
  size instead of erroring cleanly. This is the one real compatibility
  wrinkle, and it deserves emphasis: writers that adopt this CPEX are
  producing files that are conformant under the \emph{new} SIDS text but
  not the old one, so \textbf{a writer must not omit the point set
  unless it knows its downstream tool ecosystem supports this CPEX}.
  CGNS versions its SIDS compliance via \texttt{CGNSLibraryVersion\_t}
  (and, if CPEX~0049 is adopted, \texttt{CGNSMinRequiredVersion\_t}).
  If CPEX~0049 is adopted, this CPEX \textbf{must} be registered in its
  Feature-Version Registry (CPEX~0049, ``Feature-Version Registry''
  section) so that AUTO write mode raises a file's minimum required
  version whenever
  the implicit \texttt{FaceCenter} form is used, and so that
  version-aware readers reject or warn appropriately rather than
  silently misreading array sizes. If CPEX~0049 is not adopted, this
  CPEX should instead mandate a plain \texttt{CGNSLibraryVersion\_t}
  bump on write, as the minimum viable version-gating mechanism.

  \textbf{Why a major release.} This CPEX is targeted at CGNS
  \textbf{v5.0.0}, the next major release, rather than a point release
  on the v4.x line. This is not merely a labeling choice: under
  CPEX~0049's revised \texttt{cg\_open} logic, a library whose major
  version is older than the file's major version is rejected
  immediately, before any feature-compatibility check runs (CPEX~0049,
  ``Modified \texttt{cg\_open} Behavior,'' major-version safety guard).
  If this CPEX lands in v5.0.0 alongside CPEX~0049, a v4.x reader
  opening a v5.0-written file is hard-rejected at that guard and never
  reaches the point where it would have to interpret a point-set-free
  \texttt{FaceCenter} solution at all, eliminating the ``silently
  misreads the data'' failure mode entirely for that population of
  readers. This is a stronger guarantee than a minor-version bump could
  offer under the same mechanism, since CPEX~0049 only hard-rejects on
  a major-version mismatch; a same-major, higher-minor file is instead
  routed through the feature-aware check, which depends on this CPEX's
  Feature-Version Registry entry being present and correctly detected.

\item \textbf{Existing writers:} Unaffected. Codes that do not use the
  new implicit form continue to write the explicit point set exactly as
  before; nothing is deprecated or removed. \emph{Today}, this ordering
  independence is unconditional: \texttt{cg\_field\_write} branches on
  whether a point set is already attached to the solution at the moment
  \emph{it} is called (\texttt{sol->ptset == NULL}), and writing a
  \texttt{FaceCenter} field before any point set exists already fails on
  an unstructured zone (no implicit sizing rule exists yet), so no
  existing writer can depend on a field-before-point-set ordering. This
  CPEX changes that: once implicit sizing exists, a solution written
  field-first would size against the NGON face count, and a point set
  added \emph{afterward} would silently reinterpret an
  already-on-disk array written to a different, incompatible size. The
  reference implementation must therefore forbid attaching a
  \texttt{PointList}/\texttt{PointRange} to a \texttt{FlowSolution\_t}/
  \texttt{DiscreteData\_t} node that already has one or more
  \texttt{DataArray\_t} children written under implicit sizing --- returning
  \texttt{CG\_ERROR} from the point-set write call in that case --- rather
  than silently permitting the reinterpretation. This is a new ordering
  constraint introduced by this CPEX, not a preexisting one, and should be
  covered by its own test case (Section~\ref{sec:test-plan}).

\item \textbf{ABI:} No structure layout changes are required beyond the
  addition of the internal \texttt{cgi\_zone\_ngon\_face\_range()} helper
  and its public wrapper \texttt{cg\_zone\_ngon\_face\_range()}
  (Section~\ref{sec:mll_helper}). The public API changes are limited to
  widening \texttt{cg\_sol\_write}'s accepted \texttt{location} values
  (already a bug fix independent of this CPEX) and adding the new
  read-side wrapper, and are purely additive in effect
  (previously-rejected input is now accepted; no previously-accepted
  input is rejected; the wrapper is an optional convenience with no
  existing caller). \textbf{No new public error codes are introduced.}
  A write requesting implicit \texttt{FaceCenter} sizing on a zone that
  lacks complete NGON face coverage returns the existing
  \texttt{CG\_ERROR} (with a \texttt{cgi\_error()} message identifying
  the missing coverage), exactly as an out-of-range \texttt{GridLocation}
  does today; \texttt{cg\_zone\_ngon\_face\_range()} returns the same
  \texttt{CG\_OK}/\texttt{CG\_NODE\_NOT\_FOUND}/\texttt{CG\_ERROR}
  contract as every other existing \texttt{cg\_*\_info}-style read
  function, so no new return-code semantics are introduced at the public
  boundary either.

\item \textbf{Parallel I/O (\texttt{cgp\_*}):} Unaffected. The relaxation
  changes only whether a point-set child node must exist; it does not
  change the collective/independent write model for
  \texttt{DataArray\_t} contents.
\end{itemize}

\section{Examples}
\label{sec:examples}
%% ═════════════════════════════════════════════════════════════════════════

\subsection{Example 1: Before --- Identity \texttt{PointRange} (current, still valid)}
\label{sec:example1}

\begin{lstlisting}[language=C]
int S, F;
cg_sol_write(fn, B, Z, "CellData", CGNS_ENUMV(CellCenter), &S);
cg_field_write(fn, B, Z, S, CGNS_ENUMV(RealDouble), "Pressure",
               pressure, &F);

/* FaceCenter WITH an explicit point set already works today in one
 * call via cg_sol_ptset_write -- the gap this CPEX's reference
 * implementation fixes (see Existing Implementation Gaps, above) is
 * specifically the point-set-FREE form below, not this one. The
 * identity PointRange must still be kept in sync with NGonElements'
 * ElementRange by hand. */
cgsize_t range[2] = {1, 36};
cg_sol_ptset_write(fn, B, Z, "FaceData", CGNS_ENUMV(FaceCenter),
                   CGNS_ENUMV(PointRange), 2, range, &S);
cg_field_write(fn, B, Z, S, CGNS_ENUMV(RealDouble), "NormalX",
               normal_x, &F);
\end{lstlisting}

\subsection{Example 2: After --- Implicit \texttt{FaceCenter} (this CPEX)}
\label{sec:example2}

\begin{lstlisting}[language=C]
int S, F;

/* FaceCenter now accepted directly by cg_sol_write (fix described in
 * this CPEX's MLL Changes section); no PointRange needed -- the
 * zone's NGonElements section already defines the face numbering. */
cg_sol_write(fn, B, Z, "FaceData", CGNS_ENUMV(FaceCenter), &S);
cg_field_write(fn, B, Z, S, CGNS_ENUMV(RealDouble), "NormalX",
               normal_x, &F);   /* size validated against NGON face count */
\end{lstlisting}

Resulting tree (compare to the example in Section~\ref{sec:problem-statement}):

\begin{lstlisting}[language=C]
fluid_zone Zone_t [[27  8  0]]
+---ZoneType ZoneType_t "Unstructured"
+---NGonElements Elements_t [22  0]
|   +---ElementRange IndexRange_t [ 1 36]
+---NFaceElements Elements_t [23  0]
|   +---ElementRange IndexRange_t [37 44]
+---CellData FlowSolution_t
|   +---GridLocation GridLocation_t "CellCenter"
|   +---Pressure DataArray_t
+---FaceData FlowSolution_t
    +---GridLocation GridLocation_t "FaceCenter"
    +---NormalX DataArray_t     /* 36 values; indexed 1..36 against
                                    NGonElements, no PointRange node */
\end{lstlisting}

The MLL's Fortran bindings (\texttt{cg\_sol\_write\_f}, \texttt{cg\_field\_write\_f})
forward the same \texttt{location} argument through the same validation as
their C counterparts, so both the \texttt{cg\_sol\_write} allow-list fix
and this CPEX's implicit-sizing behavior are directly visible to Fortran
callers, who make up a substantial share of CGNS writers. The Fortran
mirror of Example~\ref{sec:example2}:

\begin{lstlisting}[language=Fortran]
integer :: S, F, ier

! FaceCenter now accepted directly by cg_sol_write_f; no PointRange
! needed -- the zone's NGonElements section defines the numbering.
call cg_sol_write_f(fn, B, Z, 'FaceData', FaceCenter, S, ier)
if (ier .ne. CG_OK) call cg_error_exit_f()
call cg_field_write_f(fn, B, Z, S, RealDouble, 'NormalX', &
                      normal_x, F, ier)
if (ier .ne. CG_OK) call cg_error_exit_f()
\end{lstlisting}

\subsection{Example 3: Reading --- Dispatching on Coverage}
\label{sec:example3}

\begin{lstlisting}[language=C]
char name[33];
CGNS_ENUMT(GridLocation_t) loc;
CGNS_ENUMT(PointSetType_t) ptype;
cgsize_t npnts, fstart, fend;

if (cg_sol_info(fn, B, Z, S, name, &loc)) return handle_error();

/* cg_sol_ptset_info addresses solution S directly and reports "no
 * point set" as CG_OK with ptype == PointSetTypeNull -- unlike the
 * cg_goto-positioned cg_ptset_info, it needs no prior cg_goto call,
 * and a real read failure is distinguishable from "point set absent". */
if (cg_sol_ptset_info(fn, B, Z, S, &ptype, &npnts)) return handle_error();
int has_ptset = (ptype != CGNS_ENUMV(PointSetTypeNull));

if (loc == CGNS_ENUMV(FaceCenter) && !has_ptset) {
    /* Node shape alone (FaceCenter, no point set) is necessary but
     * NOT sufficient: a pre-CPEX-0051 or non-conformant file can
     * present this same shape without complete NGON coverage. Verify
     * before trusting it -- this CPEX makes the check cheap, not
     * optional, which is why cg_zone_ngon_face_range is public. */
    if (cg_zone_ngon_face_range(fn, B, Z, &fstart, &fend) != CG_OK) {
        /* Claims the implicit form but coverage doesn't hold:
         * non-conformant file. Fail safely rather than guessing a
         * size -- this is the "silently misreads the data" failure
         * mode Section 4 warns against. */
        return handle_nonconformant_file(fn, B, Z, S);
    }
    process_all_faces(fn, B, Z, S, fend - fstart + 1);
} else if (loc == CGNS_ENUMV(FaceCenter) && has_ptset) {
    /* Explicit form: a genuine subset, or a pre-CPEX-0051 file. */
    process_face_subset(fn, B, Z, S, ptype, npnts);
}
\end{lstlisting}

\subsection{Example 4: Rejected --- Non-Contiguous NGON\_n Sections}
\label{sec:example4}

\begin{lstlisting}[language=C]
/* Zone has TWO NGON_n sections separated by an unrelated section,
 * e.g. ElementRange [1,20] (NGON_n), [21,30] (some other type),
 * [31,50] (NGON_n). Condition 2 of the coverage definition is
 * violated: the NGON_n ranges do not form a single contiguous block,
 * so there is no well-defined [E_start, E_end] to index against. The
 * explicit point set is still required.
 *
 * Note: a zone with an NGON_n section PLUS a separate boundary-patch
 * QUAD_4 section OUTSIDE the NGON_n range (e.g. ElementRange [1,36]
 * NGON_n, [37,44] QUAD_4 for BC_t) is fine and DOES qualify -- see
 * the coverage definition's revised Condition 2 and Example 5 below. */
cg_sol_write(fn, B, Z, "FaceData", CGNS_ENUMV(FaceCenter), &S);
/* cg_field_write size-validation falls back to requiring an
 * explicit PointList/PointRange, and cgnscheck flags its absence. */
\end{lstlisting}

\subsection{Example 5: Accepted --- NGON\_n Coexisting with a Boundary-Patch Section}
\label{sec:example5}

\begin{lstlisting}[language=C]
/* Zone has an NGON_n section [1,36] covering every face, PLUS a
 * separate QUAD_4 section [37,44] duplicating the boundary faces
 * for a BC_t patch -- a common pattern for tool compatibility.
 * This still qualifies for implicit FaceCenter indexing: the
 * NGON_n range [1,36] is contiguous on its own, and the QUAD_4
 * section lies entirely outside it. */
cg_sol_write(fn, B, Z, "FaceData", CGNS_ENUMV(FaceCenter), &S);
cg_field_write(fn, B, Z, S, CGNS_ENUMV(RealDouble), "NormalX",
               normal_x, &F);   /* 36 values, indexed against [1,36] */
\end{lstlisting}

\section{Test Plan}
\label{sec:test-plan}
%% ═════════════════════════════════════════════════════════════════════════

The reference implementation's test suite must cover at least the
following fixtures. Items~1--5 exercise the coverage predicate and the
new sizing rule directly; items~6--7 pin down behavior this CPEX makes
newly reachable and that would otherwise be left implicit; item~8 guards
the one existing entry point this CPEX widens.

\begin{enumerate}[leftmargin=2em]
\item Single \texttt{NGON\_n} section, implicit form
  (Example~\ref{sec:example2}): round-trip write, read, and
  \texttt{cgnscheck} clean (zero errors) in both C and Fortran (Fortran
  mirror of Example~\ref{sec:example2}).
\item Multiple contiguous \texttt{NGON\_n} sections partitioning one
  \texttt{[E\_start, E\_end]} block: accepted, sized correctly.
\item Gap case (Example~\ref{sec:example4}): write of a solution
  requesting implicit sizing is rejected with \texttt{CG\_ERROR}; a file
  constructed to have this shape externally is flagged by
  \texttt{cgnscheck} and by \texttt{cg\_zone\_ngon\_face\_range()}
  returning \texttt{CG\_ERROR} on read.
\item Boundary-patch coexistence (Example~\ref{sec:example5}): accepted.
  This is new behavior with no installed base to regress against, so it
  carries the highest regression value of any fixture here --- it was a
  deliberate relaxation from an earlier draft of this CPEX
  (Section~\ref{sec:definition}) and deserves its own dedicated test
  rather than incidental coverage from another fixture.
\item Legacy identity-\texttt{PointRange} file
  (Example~\ref{sec:example1}): still passes, unchanged size, confirming
  the explicit-point-set branch of \texttt{DataSize[]} retains precedence
  over the new implicit branch.
\item Point-set-written-after-implicit-fields ordering: attempt to attach
  a \texttt{PointRange}/\texttt{PointList} to a \texttt{FlowSolution\_t}
  that already holds fields written under implicit NGON sizing; must
  return \texttt{CG\_ERROR} per the rule in
  Section~\ref{sec:backward-compat}. This behavior is otherwise
  undefined by the prose alone; the test is what pins it down.
\item Malformed \texttt{ElementRange} inputs to
  \texttt{cgi\_zone\_ngon\_face\_range()}: \texttt{start~<~1},
  \texttt{end~<~start}, and (on a 32-bit \texttt{cgsize\_t} build) a range
  whose face count overflows subsequent size arithmetic --- all three
  must return \texttt{CG\_ERROR} before any allocation is attempted
  downstream (Section~\ref{sec:mll_helper}).
\item Structured zone with plain \texttt{FaceCenter} through the fixed
  \texttt{cg\_sol\_write} (Section~\ref{sec:mll_sol_write_fix}): must
  still be accepted for \texttt{CellDimension~$\geq$~3} (matching
  \texttt{cg\_gridlocation\_write}'s existing behavior) and rejected
  below that, pinning down the delegated-\texttt{cgi\_check\_location}
  behavior in both directions.
\end{enumerate}

\textbf{Note on \texttt{cgnscheck} regression risk.} Once
\texttt{get\_data\_size()} is fixed to honor an explicit point set
(Section~\ref{sec:cgnscheck_gap}), files that previously produced a
false-positive \texttt{ERROR:} for \emph{any} reason unrelated to
sizing will, for the first time, actually have their declared array size
checked against \texttt{ListLength[]}. Any existing file whose
\texttt{FlowSolution\_t} array size does not genuinely match its point
set's \texttt{ListLength} --- previously masked by the unconditional
unstructured-zone bailout --- will newly and correctly fail. This is the
widest legacy-system ripple in this proposal: it is a correctness fix,
not a regression, but should be called out explicitly in \texttt{cgnscheck}
release notes so it is not mistaken for one.

%% ═════════════════════════════════════════════════════════════════════════
\section{Design Decisions}
%% ═════════════════════════════════════════════════════════════════════════

\subsection{Alternative 1: Broaden Coverage to Any Complete 2-D Section}

A more general rule could drop Note~5 whenever \emph{any} face-bearing
element type's combined \texttt{ElementSize} equals the zone's total
face count, regardless of type (\texttt{TRI\_3}/\texttt{QUAD\_4}/
\texttt{NGON\_n}). \textbf{Rejected.} Nothing in the SIDS guarantees that
classical 2-D sections are complete; in the large majority of existing
files they represent boundary patches only. This alternative would
require either (a)~trusting a coincidental size match, which is fragile
and has produced silent misinterpretation bugs in similar contexts
elsewhere in the ecosystem, or (b)~introducing a new explicit
``these elements are all the faces'' marker --- which is just a
worse-ergonomics version of what \texttt{NGON\_n} already means by
definition. Scoping strictly to \texttt{NGON\_n} avoids inventing any new
machinery.

\subsection{Alternative 2: New Explicit Marker Node (e.g., \texttt{FacesComplete})}

Introduce a boolean/flag child of \texttt{Zone\_t} or \texttt{Elements\_t}
stating explicitly that a section enumerates all faces --- reviving, in
spirit, the 2005-era ``\texttt{ZoneElementType}'' idea from the original
face-based storage proposal that predated \texttt{NGON\_n}/\texttt{NFACE\_n}
standardization. \textbf{Rejected.} \texttt{NGON\_n} already carries this
meaning by SIDS definition (\S7.3); adding a redundant marker node
reintroduces exactly the kind of ``assert something already implied''
overhead this CPEX exists to remove, just relocated from
\texttt{FlowSolution\_t} to \texttt{Zone\_t}.

\subsection{Alternative 3 (chosen): Structural Inference from \texttt{NGON\_n} Coverage}

Infer completeness structurally from the conditions in
Section~\ref{sec:definition}, using only guarantees the SIDS already
makes. No new nodes; no new trust assumptions beyond ones the standard
already relies on for \texttt{CellCenter}. This is the design adopted by
this CPEX.

\subsection{On Scope: Why Not Also Amend \texttt{ZoneSubRegion\_t}?}
\label{sec:zonesubregion-scope}

\texttt{ZoneSubRegion\_t} (\S7.9) shares CPEX~0031 ancestry with
\texttt{FlowSolution\_t} and \texttt{DiscreteData\_t}, but it always
requires an explicit region descriptor by design --- a subregion that
happened to cover every face would be indistinguishable from simply
using \texttt{FlowSolution\_t}, so there is no analogous ``identity point
set'' pain point to remove. This is left as an open question for the
committee rather than folded into this proposal's scope.

\section{Open Questions}
\label{sec:open-questions}
%% ═════════════════════════════════════════════════════════════════════════

\begin{enumerate}[leftmargin=2em]
\item Should \texttt{cgnscheck} \emph{warn} (rather than silently accept)
  when it encounters the pre-CPEX-0051 identity-\texttt{PointRange}
  pattern on a complete-NGON zone, to nudge writers toward the more
  compact implicit form? This proposal takes no position; it is a
  tooling policy choice, not a conformance question.
\item Should version gating use CPEX~0049's
  \texttt{CGNSMinRequiredVersion\_t}/feature-mask mechanism, if adopted,
  or a plain \texttt{CGNSLibraryVersion\_t} bump? This CPEX assumes
  whichever mechanism is current SIDS practice at the time of adoption
  and does not mandate a specific one.
\item \label{item:cpex0050-dependency} CPEX~0050 (\texttt{DofStorage\_t}), Example~4, already writes a
  \texttt{FaceCenter} \texttt{TraceSolution} via a single
  \texttt{cg\_sol\_write} call with no accompanying point set, for a
  zone implied to have \texttt{NGON\_n} coverage. That example is only
  fully conformant once both this CPEX (for the missing point set) and
  the \texttt{cg\_sol\_write} allow-list fix
  (Section~\ref{sec:cg_sol_write_gap}) are adopted. If both CPEXes
  proceed, their reference implementations should be sequenced so
  CPEX~0051 lands first, or the dependency should be noted explicitly in
  CPEX~0050.
\end{enumerate}

\section{Summary of API Additions}
\label{sec:api-summary}
%% ═════════════════════════════════════════════════════════════════════════

The two implementation gaps from Section~\ref{sec:implementation-gaps}
(\texttt{cg\_sol\_write}'s allow-list, and \texttt{cgnscheck}'s missing
point-set read) are pre-existing bugs, independent of the NGON-coverage
logic itself, and do not need to wait on this CPEX's SIDS-level
relaxation or on \texttt{v5.0.0}. They can and should land as standalone
patches against the current release line as soon as they are reviewed;
this CPEX's reference implementation only \emph{depends on} them, so that
Example~\ref{sec:example2}'s single-call write works end to end.
